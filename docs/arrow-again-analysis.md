## 一、产品核心判断

这不是 `Parking Jam` 原方案的“挪车出库”玩法。

这份文档描述的是另一类更轻的**箭头消除 / 路径阻挡解谜**：

- 玩家点击一个箭头棋子
- 如果该箭头朝向的出口路径无遮挡
- 该棋子就会沿方向飞出棋盘并被消除
- 目标是清空全部棋子

一句话说：

**它更像“按方向和阻挡关系进行剥离式消除”的棋盘推理解谜，而不是经典 Parking Jam 那种连续拖拽移车。**

## 二、玩法结构拆解

## 1. 核心循环

单关核心循环：

1. 观察当前棋盘
2. 找出“当前可射出”的箭头棋子
3. 点击该棋子
4. 棋子飞出并消除
5. 棋盘阻挡关系变化
6. 新的可消除棋子出现
7. 重复，直到清空全盘

这本质是：

- 离散状态
- 依赖关系解锁
- 从外向内剥离
- 低操作负担，高观察与推理权重

## 2. 关键规则

### 可消除条件

- 箭头指向的出口路径上没有其他棋子阻挡
- 如果有阻挡，则该棋子当前不可点击生效

### 推理关系

文档里明确给出三类关键关系：

- 同方向剥离
- 相向阻挡
- 纵横联动

这说明该游戏的解法核心不是路径规划，而是：

**谁先解锁谁、谁阻挡谁、哪一层依赖先被移除。**

## 3. 失败系统

文档方案里有生命值 / 水滴：

- 点击错误棋子扣 1 滴
- 错误棋子标红
- 水滴归零仍可 1 星过关或按失败逻辑处理

这说明它不是纯无惩罚推理，而是带：

- 试错成本
- 错误反馈
- 星级结果

## 三、与当前 game-arrow-again 原型的关系

## 1. 相同点

可以共用的底层思想：

- 棋盘型解谜
- 离散状态更新
- 规则核可纯逻辑实现
- Android / iOS 双端共用核心层
- 广告 / 内购 / 埋点逻辑仍然成立

## 2. 不同点

当前 `game-arrow-again` 原型做的是：

- 拖拽车辆
- 轴向移动
- 位置变化驱动解法

而 `Arrow Again` 方案要求的是：

- 点击箭头
- 判断该方向路径是否畅通
- 若畅通则直接移除棋子
- 棋盘无需拖拽交互

也就是说：

**当前原型的“拖拽交互”方向，与这份文档的核心玩法并不一致。**

## 3. 结论

如果这份 PDF 是你现在真正要做的游戏定义，那么项目需要做一次明确收口：

- 当前项目已统一改名为 `game-arrow-again`，工程壳可继续沿用
- 但玩法核心应切换为：
  - `Arrow Again / 箭了又箭` 的箭头消除规则核
- 当前拖拽式 `Parking Jam` 原型不能直接当最终 MVP 继续扩写
- 它最多只能视为“之前的规则验证探索”

## 四、对技术方案的影响

## 1. 交互层变化

原先需要：

- 拖拽
- 吸附
- 移动范围预览
- 回弹

现在优先变成：

- 点击判定
- 可点击高亮
- 不可点击错误反馈
- 消除动画
- 连锁解锁的节奏反馈

## 2. 共享规则核要改写

新规则核的重点模块应改成：

- `BoardState`
- `ArrowPiece`
- `ShootabilityChecker`
- `ResolveShot`
- `MistakePenaltySystem`
- `StarScoreSystem`
- `HintSystem`

## 3. 渲染层要求更轻

由于不再需要拖拽：

- 渲染层可以更轻
- 动画重点是：
  - 点击反馈
  - 飞出动画
  - 阻挡解除后的可点击提示
  - 错误点击红色反馈

## 五、产品设计重点

## 1. 这款游戏最关键的不是手感，而是“可读性”

`Parking Jam` 更看拖拽手感。

但 `Arrow Again` 更看：

- 玩家能不能快速看懂可消除关系
- 阻挡逻辑是否一眼可理解
- 视觉复杂度会不会压垮认知
- 每一步点击是否有明确因果回报

## 2. 难度设计重点

文档里已经明确难度来自：

- 箭头数量
- 推理链深度
- 相互阻挡数
- 视觉干扰
- 特殊图案

这意味着技术上要支持：

- 纯数据驱动关卡
- 对阻挡关系的稳定计算
- 支持特殊图案布局
- 后续能做难度分层与题库管理

## 3. MVP 应重点验证

根据文档，当前 MVP 最该验证的是：

- 玩家是否 3 秒理解规则
- 是否有连续点击爽感
- 是否有“再来一关”意愿
- Hard 关是否能自然形成停留和广告点

## 六、建议的项目收口

## 1. 建议当前项目做一次明确命名层区分

建议在 `game-arrow-again` 下明确区分：

- `legacy/parking-jam-prototype`：保留现有拖拽原型参考
- `current/arrow-again-mvp`：转入当前真正要做的玩法实现

如果不想拆目录，至少也要在记录里说明：

- 当前产品定义已从“Parking Jam 拖拽出库”切到“Arrow Again 箭头剥离消除”

## 2. 建议的下一步开发方向

优先级应改成：

1. 实现箭头棋盘数据结构
2. 实现“可射出条件”判定
3. 实现点击消除与错误点击扣血
4. 实现 3–5 个引导关
5. 实现生命值 / 星级 / 重开
6. 再补动画和 Hard 关验证

## 七、一句话结论

**这份 PDF 描述的是一个比 Parking Jam 更轻、更适合点击消除式 MVP 的棋盘解谜游戏；它仍然适合 Android / iOS 双端共用规则核，但当前原型的拖拽交互方向需要及时收口，不应继续按原 Parking Jam 玩法扩写。**
